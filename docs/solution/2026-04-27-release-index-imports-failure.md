# Solution Package: Release Index Imports Failure

## Chosen Approach

Fix this hotfix at the TypeScript parser/import-extraction boundary by making `TypeScriptAdapter::extract_imports` return a deterministic, duplicate-free import fact list before indexer/storage persistence. This is the smallest adequate path because the failing storage insert is a symptom of duplicate extracted import facts/IDs, while `dh-storage` should continue enforcing strict primary-key uniqueness instead of hiding parser defects with `INSERT OR IGNORE`.

This solution depends on the approved Product Lead scope package at `docs/scope/2026-04-27-release-index-imports-failure.md` and is limited to restoring clean indexing/status for the recorded import insertion failure.

## Impacted Surfaces

- `rust-engine/crates/dh-parser/src/adapters/typescript.rs`
  - Primary fix surface: normalize/dedupe TypeScript/TSX/JavaScript/JSX import facts after extraction and before returning them to callers.
- `rust-engine/crates/dh-parser/tests/typescript_adapter.rs`
  - Add focused regression coverage for duplicate import IDs/facts from TypeScript import extraction.
- `rust-engine/crates/dh-indexer/tests/integration_test.rs`
  - Add or extend an integration fixture to prove the indexer can persist the affected TypeScript import shape without a duplicate import primary-key failure.
- `rust-engine/crates/dh-indexer/src/lib.rs`
  - Read-only/verification focus unless investigation shows indexer rewrites facts into duplicate IDs after parser extraction. Do not move the preferred fix here unless parser-level dedupe is proven insufficient.
- `rust-engine/crates/dh-storage/src/lib.rs`
  - Validation/reviewer focus only. Keep `imports.id` strict and do not change `insert_imports` to `INSERT OR IGNORE`.
- `packages/opencode-app/src/workflows/run-lane-command.ts`
  - Reproduction target only. Do not edit this file for the hotfix.

## Boundaries And Components

- Parser boundary owns normalized import fact uniqueness for each parsed source file.
- Indexer boundary may update `source_file_id` to the persisted file row but should receive a parser fact list that is already safe to insert.
- Storage boundary owns integrity enforcement and should keep surfacing duplicate primary-key errors if an upstream component emits conflicting facts.
- The hotfix must not clean, delete, regenerate, or prune release/local artifacts, including raw Semgrep JSON, `{cwd}` artifacts, `.opencode` state, generated workflow/index state, or release evidence.
- No first-run docs, release publishing, npm packaging, RC promotion, broad indexer redesign, or database schema redesign is included.

## Interfaces And Data Contracts

- `dh_types::Import.id` remains the stable primary key used by `dh-storage.imports(id INTEGER PRIMARY KEY)`.
- `LanguageAdapter::extract_imports(&self, ctx, tree) -> Vec<Import>` should return a deterministic, sorted, duplicate-free list for a single file.
- Deduplication identity should match the parser-level import fact identity, not a storage-write side effect. At minimum it must account for the fields that determine `new_import` ID material: relative path, import kind, span, raw specifier, imported name, local name, and alias. If investigation finds duplicate IDs for materially different facts, fix the parser ID material by adding the missing discriminator rather than dropping a distinct fact or masking storage errors.
- Preserve legitimate multiple import facts from a single declaration, such as default plus named imports, namespace imports, distinct named specifiers, type-only imports, side-effect imports, dynamic imports, `require(...)`, conditional requires, and re-exports.
- `dh-storage::ImportRepository::insert_imports` should remain a strict insert path. Do not use `INSERT OR IGNORE` for this issue.

## Risks And Trade-offs

- **Over-broad dedupe risk**: a naive key based only on module specifier or span could drop legitimate multiple imports from one statement. The dedupe key must include import kind and names/alias and should be covered by regression tests.
- **Under-broad dedupe risk**: sorting alone does not prevent duplicate IDs; the final returned vector must be proven unique by test.
- **Wrong-layer fix risk**: storage-level ignore would make `dh index` appear healthy while silently losing facts. That conflicts with the preferred fix and should be rejected unless explicitly re-scoped.
- **Regression scope risk**: the failing file lives in `packages/opencode-app`, but the production change should stay in `dh-parser` unless evidence proves the indexer mutates parser output into duplicates.
- **Validation cost**: the final smoke index/status commands may scan the full repository and should be run after targeted parser/indexer tests pass.

## Recommended Path

1. Reproduce the duplicate fact/ID path with the existing failure evidence and a small TypeScript parser regression fixture derived from the import shape that fails on `packages/opencode-app/src/workflows/run-lane-command.ts`.
2. Add a parser-level uniqueness assertion before the production fix so the failing behavior is pinned in `rust-engine/crates/dh-parser/tests/typescript_adapter.rs`.
3. Implement deterministic import dedupe inside `TypeScriptAdapter::extract_imports` after all import parsers have pushed facts and after/with stable sorting, so every caller receives unique import facts before storage.
4. Add/extend a `dh-indexer` integration test fixture that indexes a TypeScript file containing the duplicate-triggering import pattern and confirms indexing succeeds while persisted imports remain present and unique.
5. Keep `dh-storage` strict: no `INSERT OR IGNORE`, no schema changes, and no broad storage behavior changes.
6. Run the targeted parser/indexer tests, package-level Rust tests, then full `dh-engine` index/status smoke commands against `/Users/duypham/Code/DH`.
